# AGENTS.md

## Grounding & Judgment (最優先)

既定は no-speculation。一次ソース(公式ドキュメント/最新のコード)から確認を始め、推測で埋めない。
記憶や訓練データにある「定番の正解」から書き出さず、まず外部なら公式ドキュメントやソースコード、内部なら最新のコードを実際に読む。

- トレードオフ・好み(定説化しにくい選好)は黙って既定値を選ばず、ユーザーに選択肢を提示して確認する。
- 仮説を述べるのは「求められた時」か「必要と判断した時」のみ。述べる際は仮説だと明示し、検証手段を添える。確認できない場合は「取れない」と伝え、憶測で埋めない。
- 判断が必要な場面(トレードオフ選択、実装方針の分岐など)では、進める前にユーザーに確認する。

## 学びの記録

作業中に得た「失敗 → 原因特定 → 解決」の学びは、if-then 形式で正解だけを記録する。失敗した経緯そのもの(ナレーション)は書かない――過去の失敗を文章として残すと、未来の注意がその失敗側に歪んで再発を誘発するため。
既知のパラメータ(パス・コマンド・URL・閾値など)はリテラルで記録し、プロジェクト固有でない学びは汎化した形にする。

## ユーザーに実行させるコマンドの渡し方

ユーザー自身が実行する必要のあるコマンド(鍵情報を含む、複数行にまたがる、対話的な認証が必要、など)は、チャット本文に直接貼り付けない。
プロジェクトローカルな一時ファイル(例: `<project-root>/.claude/tmp/<task-name>.sh` 相当のパス)に書き出し、そのファイルパスを伝えてユーザーにコピーしてもらう。

理由: ターミナル直接貼り付けは改行の折り返しやMarkdownエスケープで壊れやすく、ユーザーが手動修正する必要が生じてワークフローが止まる。

適用するのは主に以下のケース:
- 対話的な資格情報入力が必要(ログイン、2FA、ブラウザ経由のOAuthコールバックなど)
- sudoパスワードなど、エージェント側から満たせないプロンプト
- Web UI操作の案内("このURLを開いて〜をクリック"等)
- ユーザーが明示的に「自分で実行する」と言った場合
- 個別承認が要る、分岐点のある長い対話的スクリプト

単一ファイルの削除や非破壊的な操作、パッケージインストールなど日常的な操作は、権限確認の仕組みに任せて直接実行してよい。

## Git ワークフロー

### コミットメッセージ形式
```
<type>: <description>

<optional body>
```
Types: feat, fix, refactor, docs, test, chore, perf, ci

### プルリクエスト作成時
1. 最新コミットだけでなく、コミット履歴全体を分析する
2. `git diff [base-branch]...HEAD` で全変更を確認する
3. 包括的なPRサマリーをドラフトする
4. テスト計画をTODO形式で含める
5. 新規ブランチは `-u` フラグ付きでpushする

### 機能実装のワークフロー
1. **先に計画する**: 依存関係とリスクを特定し、フェーズに分解してから着手する
2. **テストファースト**: テストを先に書き(RED)、実装してパスさせ(GREEN)、リファクタリングする(IMPROVE)。カバレッジ80%以上を確認する
3. **コードレビュー**: コード変更直後に自己レビュー(またはレビュー専用の別セッション)を通す。CRITICAL/HIGHの指摘は必ず対応し、MEDIUMも可能な範囲で対応する
4. **コミット&プッシュ**: 詳細なコミットメッセージ、Conventional Commits形式に従う

## セキュリティ

### コミット前の必須チェック
- [ ] ハードコードされたシークレット(APIキー・パスワード・トークン)がない
- [ ] ユーザー入力はすべてバリデーションされている
- [ ] SQLインジェクション対策(パラメータ化クエリ)
- [ ] XSS対策(HTMLのサニタイズ)
- [ ] CSRF対策が有効
- [ ] 認証/認可が検証済み
- [ ] 全エンドポイントにレート制限
- [ ] エラーメッセージが機微情報を漏らさない

### シークレット管理
シークレットはハードコードせず環境変数から読み、値が無ければ明示的にエラーを出す。

```typescript
// NG: ハードコード
const apiKey = "sk-proj-xxxxx"

// OK: 環境変数
const apiKey = process.env.OPENAI_API_KEY
if (!apiKey) {
  throw new Error('OPENAI_API_KEY not configured')
}
```

### セキュリティ問題発見時の対応
1. 直ちに作業を止める
2. 徹底したセキュリティレビューを行う
3. CRITICAL指摘を先に修正してから続行する
4. 露出したシークレットはローテーションする
5. 同種の問題が他にないかコードベース全体を確認する

## コーディングスタイル

### Simplicity First(最優先)
問題を解く最小のコードにする。
- 求められていない機能・単一用途の抽象・要求外の柔軟性/設定可能性を足さない
- 現実に起きない失敗のエラー処理を書かない。実際に起きうる失敗(外部I/O・パース・ネットワーク)だけ扱う
- 既存コードを触るときは必要な行だけ変更する。隣接の「改善」や無関係なリファクタ・整形はしない。掃除は自分の変更が生んだ未使用コードだけ
- 無関係な問題(UI崩れ・lint・テスト失敗・flaky)に気づいても黙って直さない。自分の差分には含めず、発見として報告する(直すかはユーザーが判断)
- 200行が50行になるなら書き直す。「シニアエンジニアが overcomplicated と言うか?」→ Yesなら簡素化する

### 技術判断
技術判断では開発コスト(実装労力)を大きく重み付けしない。品質・堅牢性・スケーラビリティ・長期保守性を優先する。ただし Simplicity First に従属し、品質優先を理由に投機的な抽象や要求外の柔軟性を足さない。

### Immutability
既存オブジェクトを変更(mutate)せず、常に新しい値を作る。

### ファイル構成
少数の巨大ファイルより多数の小さなファイル。高凝集・低結合。目安200-400行、最大800行。種類別でなく機能/ドメイン別に整理する。

### 入力バリデーション
外部/信頼できない入力は検証する。内部・起こり得ない入力にはバリデーションを足さない(Simplicity First優先)。

### コード品質チェックリスト(作業完了前)
- [ ] 読みやすく命名が適切
- [ ] 関数が小さい(50行未満)
- [ ] ファイルが焦点を絞っている(800行未満)
- [ ] 深いネストがない(4階層超えない)
- [ ] 現実に起きる失敗だけ処理している
- [ ] デバッグ用print/console.logが残っていない
- [ ] ハードコード値がない
- [ ] mutationがない(immutableなパターン)

## テスト

### 最低カバレッジ: 80%
必須のテスト種別: Unit / Integration / E2E

### テスト駆動開発(必須ワークフロー)
1. テストを先に書く(RED)
2. 実行して失敗を確認する
3. 最小限の実装をする(GREEN)
4. 実行してパスを確認する
5. リファクタリングする(IMPROVE)
6. カバレッジを確認する(80%以上)

### バグ修正のワークフロー
バグ修正は、エンドユーザーの体験に可能な限り近いE2E設定でバグを再現することから始める。再現できて初めて真因に当たっていることが保証される。再現手順を再現テスト(RED)に落とし、テスト駆動のフローに接続する。

### テスト失敗時のトラブルシューティング
1. テストの独立性を確認する
2. モックが正しいか確認する
3. (テスト自体が誤っている場合を除き)実装を直す。テストを直すのではない

## パターン: 実績あるベースを使う

新機能を実装する際は、実績のあるスケルトンプロジェクト(battle-tested)がないか探す。複数の候補がある場合は、セキュリティ・拡張性・関連性・実装計画の観点で並行して評価してから、最適なものをベースにクローンし、実績ある構造の中で反復する。

## エージェント運用の一般原則

- 複雑な機能実装に着手する前に、明示的な実装計画を立てる(依存関係とリスクの洗い出し、フェーズ分解)
- コードを書いた/変更した直後には、レビューの目(自己レビューまたは別の視点)を通す
- 行き詰まった時やアーキテクチャ判断が必要な時は、一段強いレビュー・セカンドオピニオンを得る
- 独立して並行実行できるタスクは、可能な範囲で並行に処理する
- 多数のサブエージェント/並列プロセスを一斉に生む機能(大規模fan-out)を使う前に、規模・目的・コストの概算とトレードオフを説明してユーザーの明示承認を得る。少数(1〜3体)の通常委譲は承認不要

## パフォーマンス

### コンテキストウィンドウ管理
大規模リファクタリング・複数ファイルにまたがる機能実装・複雑な相互作用のデバッグでは、コンテキストウィンドウの残り20%を使い切る手前で区切りをつける。単一ファイル編集・独立したユーティリティ作成・ドキュメント更新・単純なバグ修正はコンテキスト感度が低い。

### ビルドのトラブルシューティング
ビルド失敗時は、エラーメッセージを分析し、段階的に修正し、各修正後に検証する。
