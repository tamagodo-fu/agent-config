# Memory Writing

作業中に得たどんな「失敗 → 原因特定 → 解決」の学びも、**ユーザー指示を待たず**にプロジェクトメモリへ if-then 形式で保存してからセッションを終える。

## 書き方ルール

- **正解 if-then だけを書く**。「○○と間違えた」「最初は××だった」等の失敗ナレーションは書かない。失敗内容を文章として残すと未来の attention が失敗側に歪んで再発を呼ぶ
- 形式: `if <状況> then <行動>` の表 or 短い参照リスト
- 既知パラメータ(パス・コマンド・URL・閾値)はリテラルで書く
- 適切に**抽象化**する。プロジェクト固有でない学びはより汎化された if-then に
- 訂正/失敗ログ/反省文セクションは作らない

## トリガ(以下のどれかで即書きする)

- ユーザーから訂正・叱責を受けた
- 自分の実装/出力ミスに気づいた
- 環境(設定ファイルパス・CLI フラグ・API挙動)の正解を新たに把握した
- 3rd party tool の status / 仕様を確認できた
- セッション終了直前(その日の学びを書き出す)

## 既存メモリの保守

過去 memory に失敗ナレーションが含まれていたら、見つけ次第:
1. 正解 if-then だけ抽出
2. 失敗説明部分を削除
3. タイトル・description も「失敗○○」→「正解○○」に書き換え

詳細は各プロジェクトの `memory/feedback_memory_writing_style.md` (存在する場合) を参照。無い場合は本ルールに従って新規作成する。
